# DeepWiki: Update Wiki

Update a wiki previously generated by `/deepwiki:wiki` so it reflects the repository's current state. The original DeepWiki had no incremental path — its "Refresh Wiki" button deleted the server-side cache and re-ran the whole pipeline. This skill adds the incremental path in the wrapper layer: it keeps the existing `.deepwiki/structure.xml`, regenerates only the pages whose source files changed, and fills the same verbatim templates in `${CLAUDE_PLUGIN_ROOT}/templates/` — no prompt text is added or altered.

## Arguments

`$ARGUMENTS` may contain, in any order:
- A git ref (commit, tag, or branch) to diff against, overriding the recorded baseline
- The word `restructure` to force re-determination of the wiki structure (Step 5)

## Step 1 — Load the existing wiki

1. Require `.deepwiki/structure.xml` and at least one page in `.deepwiki/pages/`. If either is missing, tell the user to run `/deepwiki:wiki` first and stop.
2. Parse the structure XML: every `<page>` with its `id`, `<title>`, and `<relevant_files>`, plus `<sections>` when present.
3. Recover the generation settings, preferring `.deepwiki/metadata.json` when it exists (`view`, `language`, `commit`). Fallbacks for wikis generated before metadata existed:
   - view: `comprehensive` if the XML has `<sections>`, else `concise`
   - language code: infer from the language the existing pages are written in; resolve `${language_name}` with the mapping table in `${CLAUDE_PLUGIN_ROOT}/skills/wiki/SKILL.md`
4. Choose the diff baseline — the first of these that names a commit that actually exists (verify with `git rev-parse --verify <ref>^{commit}`; on failure fall through to the next):
   1. a git ref passed in `$ARGUMENTS`
   2. the `commit` recorded in `.deepwiki/metadata.json`
   3. the last commit that touched the wiki: `git log -1 --format=%H -- .deepwiki`
   If none works (not a git repository, shallow clone, rewritten history), there is no baseline: treat **every** page as stale — Step 4 then regenerates all pages against the existing structure — and say so in the report.

## Step 2 — Compute the change set

1. Committed changes since the baseline: `git diff --name-status -M <baseline>..HEAD`
2. Uncommitted changes: `git status --porcelain`
3. Merge both, ignoring paths under `.deepwiki/`. Classify each path as added, modified, deleted, or renamed (old path → new path).
4. If the set is empty, report that the wiki is already up to date and stop.

## Step 3 — Map changes to pages

1. Mark a page **stale** if any path in its `<relevant_files>` was modified, deleted, or renamed.
2. Fix up each page's file list: drop deleted files, replace renamed paths with their new path.
3. Changed files that appear in no page's `<relevant_files>` are **orphans**. Assign each orphan to the page(s) it most plausibly belongs to — judge by directory, imports, naming, and each page's title and description — append it to that page's file list and mark the page stale. Files no wiki page should cover (lockfiles, CI configuration, binary assets) may be left unassigned; list them in the report.
4. Escalation check — an incremental pass gives a poor result when the wiki's shape itself is stale. If any of these hold:
   - a top-level source directory was added or removed, or the README was substantially rewritten
   - the orphans form a coherent new subsystem outside every existing page's scope
   - a page has lost all of its relevant files
   - more than half of all pages are stale

   and `restructure` was not passed, ask the user whether to restructure (Step 5) or continue incrementally; if asking is not possible, continue incrementally and put the recommendation in the report.

## Step 4 — Regenerate stale pages (incremental path)

Skip this step when restructuring. This is exactly the wiki skill's Step 3, applied only to stale pages:

1. Fill `${CLAUDE_PLUGIN_ROOT}/templates/wiki-page.md` per stale page: `${page.title}` (all occurrences), `${language_name}`, and `${file_list}` built from the page's **updated** file list, one `- [path/to/file](../../path/to/file)` line per file.
2. Launch a `deepwiki:page-writer` agent per page — filled prompt, then `Other wiki pages:` (one `<page-id>: <title>` line per other page in the structure), then `Output file:` with the absolute path to `.deepwiki/pages/<page-id>.md`. Run in parallel batches of 3–5; re-run a failed page once.
3. Write the updated `<relevant_files>` back into `.deepwiki/structure.xml` (orphan additions, deletions, renames) so future updates track them. Change nothing else in the XML.

## Step 5 — Restructure (only when requested or confirmed)

Re-determines the structure while preserving pages that did not change:

1. Rebuild the wiki skill's Step 1 inputs (fresh `${fileTree}` and `${readme}`) and execute the structure template for the recovered view, producing a new `<wiki_structure>` XML.
2. Reconcile it with the old structure: when a new page clearly corresponds to an existing page (same topic/title), rewrite its `id` in the new XML to the existing page's id, so file names and cross-page links keep working.
3. A carried-over page keeps its existing Markdown untouched only if its `<relevant_files>` are unchanged (after rename fixups) and none of them appear in the change set. Every other page — new, changed files, or changed scope — is generated as in Step 4.
4. Save the reconciled XML to `.deepwiki/structure.xml`; delete files in `.deepwiki/pages/` whose page ids no longer appear in it.

## Step 6 — Refresh index and metadata

1. Rebuild `.deepwiki/README.md` exactly as the wiki skill's Step 4 specifies, with the current timestamp.
2. Write `.deepwiki/metadata.json`:

```json
{
  "view": "<comprehensive|concise>",
  "language": "<language code>",
  "commit": "<git rev-parse HEAD, or null if not a git repository>",
  "dirty": <true if git status --porcelain lists changes outside .deepwiki/>,
  "generated_at": "<same timestamp as the index>"
}
```

## Step 7 — Report

Tell the user: the baseline used (and how it was chosen); the change-set counts (added/modified/deleted/renamed) and any unassigned orphans; which pages were regenerated, kept, added, or removed (with paths); any page that failed; and, if the escalation check fired but no restructure ran, the recommendation to run `/deepwiki:update restructure`. Do not print page contents in the conversation.
