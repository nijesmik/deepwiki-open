# DeepWiki: Update Wiki

Update a wiki previously generated by `/deepwiki:wiki` so it reflects the repository's current state. The original DeepWiki had no incremental path — its "Refresh Wiki" button deleted the server-side cache and re-ran the whole pipeline. This skill adds the incremental path in the wrapper layer: it keeps the existing `.deepwiki/structure.xml`, regenerates only the pages whose source files changed, and fills the same verbatim templates in `${CLAUDE_PLUGIN_ROOT}/templates/` — no prompt text is added or altered.

## Arguments

`$ARGUMENTS` may contain, in any order:
- A git ref (commit, tag, or branch) to diff against, overriding the recorded baseline
- The word `restructure` to force re-determination of the wiki structure (Step 5)

## Step 1 — Load the existing wiki

1. Require `.deepwiki/structure.xml` and at least one page in `.deepwiki/pages/`. If either is missing, tell the user to run `/deepwiki:wiki` first and stop.
2. Parse the structure XML: every `<page>` with its `id`, `<title>`, and `<relevant_files>`, plus `<sections>` when present.
3. Recover the generation settings, preferring `.deepwiki/metadata.json` when it exists (`view`, `language`, `commit`). Fallbacks for wikis generated before metadata existed:
   - view: `comprehensive` if the XML has `<sections>`, else `concise`
   - language code: infer from the language the existing pages are written in; resolve `${language_name}` with the mapping table in `${CLAUDE_PLUGIN_ROOT}/skills/wiki/SKILL.md`
4. Choose the diff baseline — try these in order, verifying each candidate with `git rev-parse --verify <ref>^{commit}`:
   1. a git ref passed in `$ARGUMENTS`. If it fails verification, stop with an error — the user named it explicitly, and silently falling back would diff against a baseline they did not choose.
   2. the `commit` recorded in `.deepwiki/metadata.json`. If the metadata says `"dirty": true`, this baseline is imprecise — the wiki already reflects changes that were uncommitted at generation time, so some pages may be regenerated needlessly; note this in the report.
   3. the last commit that touched the wiki: `git log -1 --format=%H -- .deepwiki`
   When an automatic candidate (2 or 3) fails verification (shallow clone, rewritten history, wiki never committed), fall through to the next. If none works, or this is not a git repository at all, there is no baseline. The only sound option is regenerating every page against the existing structure — which is also the most expensive one — so ask the user before proceeding (re-running `/deepwiki:wiki` is the full equivalent); if asking is not possible, proceed and say so in the report. In this no-baseline mode, skip Steps 2 and 3 entirely: every page is stale, with its file list exactly as the structure XML has it.

## Step 2 — Compute the change set

1. Committed changes since the baseline: `git diff --name-status -M <baseline>..HEAD`
2. Uncommitted changes: `git status --porcelain`
3. Merge both, ignoring paths under `.deepwiki/`. Classify each path as added, modified, deleted, or renamed (old path → new path). `git status --porcelain` does not pair unstaged renames — treat a deleted tracked path plus a near-identical untracked path as a rename.
4. If the set is empty: when `restructure` was passed, go directly to Step 5 (a forced restructure needs no change set); otherwise backfill `.deepwiki/metadata.json` per Step 6 if it is missing, report that the wiki is already up to date, and stop.

## Step 3 — Map changes to pages

1. Mark a page **stale** if any path in its `<relevant_files>` was added, modified, deleted, or renamed. (Added paths can legitimately appear in the list — the wiki skill's file tree includes files that were still untracked at generation time.)
2. Fix up each page's file list: drop deleted files, replace renamed paths with their new path.
3. Changed files that appear in no page's `<relevant_files>` are **orphans**. Assign each orphan to the page(s) it most plausibly belongs to — judge by directory, imports, naming, and each page's title and description — append it to that page's file list and mark the page stale. Files no wiki page should cover (lockfiles, CI configuration, binary assets) may be left unassigned; list them in the report.
4. Escalation check — an incremental pass gives a poor result when the wiki's shape itself is stale. If any of these hold:
   - a top-level source directory was added or removed, or the README was substantially rewritten
   - the orphans form a coherent new subsystem outside every existing page's scope
   - a page has lost all of its relevant files
   - more than half of all pages are stale

   and `restructure` was not passed, ask the user whether to restructure (Step 5) or continue incrementally; if asking is not possible, continue incrementally and put the recommendation in the report.

## Step 4 — Regenerate stale pages (incremental path)

Skip this step when restructuring. This is exactly the wiki skill's "Step 3 — Generate each page", applied only to stale pages, with one exclusion: a stale page whose file list became empty after the fixups is **not** regenerated — a page-writer forced to find sources for a deleted subsystem would confabulate. Keep its Markdown as-is and flag it in the report as a removal candidate (removing pages requires a restructure).

1. Fill `${CLAUDE_PLUGIN_ROOT}/templates/wiki-page.md` per stale page: `${page.title}` (all occurrences), `${language_name}`, and `${file_list}` built from the page's **updated** file list, one `- [path/to/file](../../path/to/file)` line per file.
2. Launch a `deepwiki:page-writer` agent per page — filled prompt, then `Other wiki pages:` (one `<page-id>: <title>` line per other page in the structure), then `Output file:` with the absolute path to `.deepwiki/pages/<page-id>.md`. Run in parallel batches of 3–5; re-run a failed page once.
3. Write the updated `<relevant_files>` back into `.deepwiki/structure.xml` (orphan additions, deletions, renames) so future updates track them. Change nothing else in the XML.

## Step 5 — Restructure (only when requested or confirmed)

Re-determines the structure while preserving pages that did not change:

1. Rebuild the inputs from the wiki skill's "Step 1 — Gather repository inputs" (`${owner}`, `${repo}`, fresh `${fileTree}` and `${readme}`) and execute its "Step 2 — Determine the wiki structure" for the recovered view and `${language_name}`, producing a new `<wiki_structure>` XML — every `file_path` must exist in the repository, as that step requires.
2. Reconcile it with the old structure: when a new page clearly corresponds to an existing page (same topic/title), rewrite its `id` in the new XML to the existing page's id, so file names and cross-page links keep working. Both runs typically number pages `page-1…page-N`, so id collisions are the norm: before reusing an id, move any *different* new page currently holding it to a fresh unused id; apply every id rewrite to all `<page_ref>` and `<related>` occurrences as well; save only after checking that all ids are unique and every reference resolves.
3. A carried-over page keeps its existing Markdown untouched only if none of its old files appear in the change set and its scope did not change — meaning the new XML's `<relevant_files>` equal, as a set, the old page's fixed-up list. The fresh run re-picks files nondeterministically, so for pages that clearly correspond you may copy the old fixed-up list into the new XML to preserve them. Every other page — new, changed files, or changed scope — is generated as in Step 4.
4. Save the reconciled XML to `.deepwiki/structure.xml`; delete files in `.deepwiki/pages/` whose page ids no longer appear in it, and also regenerate any kept page whose Markdown links to a deleted page's `<page-id>.md`.

## Step 6 — Refresh index and metadata

1. Rebuild `.deepwiki/README.md` exactly as the wiki skill's "Step 4 — Build the index" specifies, with the current timestamp. Take `<repo_url>` from the first line of the existing index (or re-derive it as wiki Step 1 does).
2. Write `.deepwiki/metadata.json` — this schema must stay identical to the block in the wiki skill:

```json
{
  "view": "<comprehensive|concise>",
  "language": "<language code>",
  "commit": "<git rev-parse HEAD, or null if not a git repository>",
  "dirty": <true if git status --porcelain lists changes outside .deepwiki/; false if not a git repository>,
  "generated_at": "<same timestamp as the index>"
}
```

## Step 7 — Report

Tell the user: the baseline used (and how it was chosen, including the dirty-baseline caveat when it applies); the change-set counts (added/modified/deleted/renamed) and any unassigned orphans; which pages were regenerated, kept, added, or removed (with paths), plus any empty-file-list pages kept as removal candidates; any page that failed; and, if the escalation check fired but no restructure ran, the recommendation to run `/deepwiki:update restructure`. Do not print page contents in the conversation.
